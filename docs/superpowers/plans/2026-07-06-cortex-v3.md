# Cortex v3 — Research + Implementation Plan

Date: 2026-07-06. Status: approved for assignment. Supersedes v2 plan (`2026-04-23-cortex-v2.md`).

## Context

Cortex v2 is a working graph-aware context engine (stdlib-only Python 3.11+, SQLite, layers STRUCTURAL/COCHANGE/HEADING/SEMANTIC) but: integration into Claude Code/Codex is fragile string-injection; structural analysis is Python-only; ranking is BFS-hop bonuses (field standard is PageRank — RepoGraph shows +32.8% SWE-bench relative from graph context); no MCP server while the 2026 field converged on MCP as agent transport; graphify duplicates its core value; several shipped bugs (broken install instructions, no-op flags, false heading sections).

Goal: revamp Cortex into a plug-and-play plugin for **both Claude Code and Codex**, absorb graphify's useful features, fix ecosystem overlaps with tokenslim/graphify, refresh all docs.

Decisions locked with user: absorb graphify features into Cortex; tree-sitter as optional extra; MCP server + plugin manifests replace string-injection; small cross-repo fixes allowed (tokenslim versions, global CLAUDE.md graphify pointer). Delegation: codex-rescue for bounded implementation slices; haiku/sonnet subagents for explore/review.

## Research: prior art

### Papers / systems
- **Aider RepoMap** — tree-sitter tags → graph → PageRank-ranked repo map under token budget; signatures-first packing. Gold standard.
- **RepoGraph (ICLR 2025)** — line-level repo code graph; +32.8% relative on SWE-bench bolted onto agents.
- **CodexGraph (NAACL 2025)** — code graph in graph DB; agent queries it iteratively instead of receiving pre-baked context.
- **LocAgent (ACL 2025)** — heterogeneous directed graph, multi-hop code localization.
- **GraphCoder (ASE 2024)**, **KGCompass** (issue↔code linking, 58.3% SWE-bench Lite), **CodeRAG** (bigraph), **Codebase-Memory (2026)** (tree-sitter KG over MCP — closest to Cortex v3 target shape), **LARGER (2026)**, RACG survey (2025).
- 2026 landscape: converged on "graphs + symbols + semantics + packing over MCP transport."

### Competing tools
Serena (LSP symbol tools, MCP), claude-context (vector semantic search, needs embeddings/vector DB), code-graph-rag (tree-sitter, 12 langs, MCP), code-review-graph (local-first persistent map), repomix (dumb packing).

### Cortex differentiators (keep + amplify)
- COCHANGE layer (git temporal coupling) — none of the above have it.
- Zero hard deps, SQLite local-first, deterministic core, LLM enrichment optional + cached.
- Communities/god-node architecture report.

### Integration surfaces (verified)
- Claude Code plugin: `.claude-plugin/plugin.json` + `.mcp.json` (auto-discovered) + skills/ + hooks + marketplace.json.
- Codex CLI 2026: skills (agentskills.io SKILL.md, `~/.codex/skills` or `.codex/skills`) shipped Dec 2025; custom prompts deprecated Jan 2026; MCP via `[mcp_servers.*]` in `~/.codex/config.toml` or `.codex-plugin/plugin.json` w/ `skills:`/`mcpServers:` keys; hooks May 2026.
- → Plug-and-play both = one stdio MCP server + dual-registered skill + plugin manifests.

## Ecosystem findings

### tokenslim — complementary; minor conflicts
Reactive tool-result compression (PostToolUse Bash/Read/Grep|Glob) vs Cortex's proactive selection — they stack. Conflicts: >40KB head/tail truncation could clip `cortex bundle` stdout via Bash (MCP path immune — matchers don't touch MCP tool names); `/slim-memory` reflows injected CLAUDE.md sections; manifest version drift (claude 0.1.0 vs codex 0.1.1). Live-observed bug: read-dedup stub claimed "content already in context" across subagent boundary when it wasn't.

### caveman — orthogonal; packaging reference
Output-side compression. Reference layout: `.claude-plugin/plugin.json` (inline hooks OK), `.codex-plugin/` + `.codex/{hooks.json,config.toml}`, `commands/`, `skills/*/SKILL.md`, `agents/*.md` (frontmatter tools+model), evals/, install.sh/ps1.

### graphify — biggest overlap → absorbed
Actually at `~/.agents/skills/graphify/SKILL.md` (global CLAUDE.md pointer to `~/.claude/skills/graphify` is stale — file doesn't exist). Wraps pip pkg `graphifyy`; duplicates repo→graph→budgeted retrieval, git hooks, CLAUDE.md injection, MCP. Resolution: Cortex owns code repos; port HTML viewer/Obsidian export/watch into Cortex (P5); graphify retained for non-code media only.

### P1a — Ranking + report quality
Files: new `src/cortex/rank.py`; `bundle.py`, `community.py`, `report.py`, `cli.py`.
- Pure-Python personalized PageRank (power iteration, weighted edges, dangling handling); personalization = normalized keyword-seed scores. Default ranker; `--rank {pagerank,bfs}` escape hatch (keep `_bfs_proximity`).
- Layer-aware transition weights: per-layer multipliers before walk (STRUCTURAL > COCHANGE > HEADING); commit nodes excluded from ranked output (provenance-only) so they don't act as hub polluters.
- Community fix: per-layer edge-weight thresholds (exclude HEADING contains-edges from propagation; COCHANGE weight floor) → kill mega-community.
- Report: filter src↔test co-change pairs from "surprising connections" (`--include-test-pairs` opt-in).
- Tests: `test_rank.py` (toy-graph convergence, dangling, empty); bundle surfaces 3-hop dep under pagerank not bfs; community no longer collapses; report filter.

### P1b — Symbol granularity + skeleton packing
Files: `store.py`, `models.py`, `ast_extract.py`, `bundle.py`.
- Schema via existing `_migrate_existing_schema()` ALTER-ADD (non-destructive): `graph_nodes` + `granularity TEXT DEFAULT 'file'`, `signature TEXT`, `span_start/span_end INTEGER`.
- Node ids `symbol:{path}:{qualname}`; ast walk emits function/class nodes w/ `ast.unparse` signatures + `contains` edges (layer STRUCTURAL).
- Skeleton packing: budget-tight file → imports + all signatures, full bodies only for top-scoring symbols (`_render_skeleton`). Skeletons visibly marked (`[skeleton: bodies elided]`) + carry span + content hash so staleness detectable.
- Regression guard: file-level report/community output unchanged immediately after P1b.
- Tests: symbol extraction, model round-trip w/ defaults, skeleton tight/loose budgets, old-DB opens cleanly.

### P2 — MCP server (stdlib stdio)
Files: new `src/cortex/mcp/{__init__,server,tools}.py`, new `src/cortex/impact.py`; `store.py`, `cli.py`.
- Newline-delimited JSON-RPC 2.0; HARD RULE stdout=protocol only, logs→stderr; graceful EOF/BrokenPipe.
- Full MCP lifecycle: `initialize` returns negotiated `protocolVersion`, `capabilities: {tools:{}}`, `serverInfo`; accept `notifications/initialized`; methods `tools/list`, `tools/call`; errors as `isError:true` content, never uncaught.
- Tools: `cortex_query` (task→budgeted bundle), `cortex_overview`, `cortex_impact` (cochange+structural neighbors ranked by weight), `cortex_search_symbols` (`store.search_nodes` LIKE query), `cortex_refresh`. Each bundle/impact item includes "why" provenance — contributing edges/path from seed (CodexGraph-style explainability).
- `repo_path` arg optional, default server cwd → `gitutils.discover_repo_root()`. Missing DB → structured error directing to `cortex_refresh`.
- Staleness: fingerprint = hash of sorted (path, size, mtime) file-set — catches deletes/renames; stored at ingest (`repos.fingerprint`); queries compare cheaply, never block — return `stale: true` + refresh hint.
- `cortex mcp` CLI subcommand (execs `python -m cortex.mcp.server`).
- Tests: subprocess round-trip initialize/list/call; EVERY stdout line must parse as JSON; impact + fingerprint units.
- Rationale: MCP tool results bypass tokenslim PostToolUse matchers (verified live).

## Key risks
| Risk | Mitigation |
|---|---|
| Hand-rolled MCP framing breaks (stray stdout) | every-stdout-line-is-JSON test; stderr-only logging by construction |
| Pure-Python PageRank perf on large graphs | iteration cap + `--rank bfs` fallback + benchmark comparison |
| tree-sitter grammar churn | isolated backend module; per-file degrade to regex |
| No-CDN viewer underspecified | P5 pre-step reads graphify source first |
| `cortex` not on PATH in host launch env | document `python -m cortex.mcp.server` fallback |
| Removing `install` breaks external refs | `cortex migrate` + prominent README migration section |
