# Changelog

## Unreleased

- Fix PageRank bundle ordering jitter across `PYTHONHASHSEED` by preserving deterministic graph and floating-point accumulation order.

- Add the Claude Code-only `cortex-explorer` read-only agent for multi-step Cortex-native repository exploration, with documented delegation guidance, Qt signal/slot graph tracing, and findings/IDs/spans/next-calls return contract.

- Add the confidence-tiered dead-code report (`cortex_dead_code`): it analyzes persisted symbol-graph fan-in plus local grep references, excludes Python entry points and test/re-exported/decorated symbols, conservatively caps regex-backend languages at low confidence, and credits Qt connects/emits/handles/instantiates edges plus Qt meta-object macros. The `cortex report` output includes deterministic dead-code candidates, and the MCP surface supports budgeted findings.

- Add diff-aware local risk analysis (`cortex risk` / `cortex_risk`): safely parses deterministic numstat, rename/status, and zero-context git diff data for committed ranges or staged changes; reports additions/deletions/churn, stored hotspot metadata, structural fan-in, fixed normalized 0–10 scores, and deterministic missing co-change/test/Qt/build-reference directives. Qt advice is emitted only from resolved graph edges and metadata, while missing-index, shallow-history, no-commit, and non-git cases remain clearly labeled. The MCP tool performs one shared freshness check, uses the standard `_meta` envelope and token ledger, and applies deterministic budget truncation without query caching.

- Remove the P1-8 fail-open `PreToolUse` redirect hook for built-in `Read`/`Grep`/`Glob` calls (added and then reverted on this branch): shell-command/built-in-tool interception is out of scope for Cortex's repo-indexing functionality and now lives in its own plan (`RTK_PLAN.md`) for a possible separate plugin.

- Add the strictly optional `[semantic]` extra for local Model2Vec static retrieval. `cortex semantic setup` is the only download path for the verified `minishlab/potion-code-16M` model; runtime ingest/query load only the managed local cache under `CORTEX_DATA_DIR`, persist float32 symbol-span chunks in `chunk_embeddings`, and fail soft to the unchanged lexical/graph path when unavailable. Semantic vectors participate as one deterministic RRF list, detailed `cortex_overview` reports local semantic status, and the eval harness has opt-in vocabulary-gap/Qt click tasks that never claim semantic-on success without a real local model.

- Add the batched `cortex_context` MCP tool (P1-1): pass ordered paths, file node ids, or symbol names in `targets` and receive one deterministic compact triage card per input under the default 2000-token budget. Exact file resolution runs before `_resolve_symbol`; ambiguous symbols return non-error disambiguation cards in the existing read-symbol match shape, and missing targets do not abort the batch. Cards include spans, signatures/Markdown headings, top structural neighbors, weighted co-change partners, hotspot data, per-card truncation, and optional `impact`/`cochange`/`symbols` expansions; target source bodies are fetched lazily once per unique resolved path, and irreducibly tiny budgets report feasibility without changing input targets. The shared `_meta` envelope and P0-1 ledger record one auditable raw-file baseline for the whole call, with exactly one freshness check. Qt cards expose QObject signals, slots, emits/connects wiring, QML handlers, and instantiation relations. Qt graph extraction now resolves locally declared C++ QObject types instantiated from QML to real class symbols in both regex and tree-sitter-parity paths, including incremental refreshes; the shared `qt_app` fixture demonstrates a real `DeviceManager` QML instantiation.
- Add deterministic hotspot analytics (P1-2): git commit touch counts plus per-language branch complexity (including C++ switch/case and QML handlers/property bindings) are persisted as `graph_nodes.metadata_json.hotspot` during full and incremental ingest. `cortex_overview` exposes `top_hotspots`, reports include a Hotspots section, and `cortex_impact` neighbors carry churn/complexity/score fields. Bundle ranking remains unchanged by default; `hotspot_boost: true` on `cortex_query` (or `cortex bundle --hotspot-boost`) opts into the bounded signal. Comment, string, triple-quoted, and C++ raw-string masking is shared with the regex structural backend, with Qt fixture coverage for a fat switch and QML handlers/bindings.
- Add aggressive read modes for read tools (P1-6): `cortex_read_symbol` gains a `mode: "full" | "skeleton" | "signature"` parameter (default `"full"`, byte-identical to the pre-P1-6 payload when omitted) — `"skeleton"` renders the symbol's own signature plus, for a class/component, its nested members' signatures with bodies elided, scoped via a new `bundle.py::_render_symbol_skeleton`; `"signature"` returns just the signature line and span metadata with no body at all. New MCP tool `cortex_read_file(path, mode="skeleton", budget=...)` is the direct raw-`Read` replacement: it renders an indexed file's import/include lines plus every top-level symbol's signature with bodies elided (reusing bundle packing's existing `_render_skeleton`), falling back transparently to full content when the file has no indexed symbols (prose/Markdown); `mode: "full"` returns the exact indexed content. Both new surfaces route through the P1-5 `_meta` envelope and the P0-1 ledger (`_estimate_baseline` now covers `cortex_read_file` and documents that `cortex_read_symbol`'s baseline is always the full raw file regardless of mode, so a cheaper mode shows a bigger `saved_tokens`). Two new `CortexStore` methods support this without a whole-repo scan: `fetch_source_record` (single-file content+kind) and `fetch_symbols_for_path` (a file's indexed symbol nodes). **Qt parity**: `bundle.py::_render_skeleton` is rewritten to nest children by *span containment* instead of dotted-qualname matching — the regex/tree-sitter structural backend never dot-qualifies C++/QML member names (a Qt header's `signals:`/`slots:` members and a QML component's `signal`/`onFoo:` handlers all get flat `symbol:<path>:<name>` ids), so the old qualname-only scheme left them stranded as spurious top-level entries instead of nested under their class/component; span containment recovers correct nesting for both conventions. The renderer now also preserves `Q_OBJECT` and Qt `signals:`/`slots:` section markers (which have no symbol node of their own) and QML `id:` property lines verbatim inside a class/component body, and elides a QML `onFoo: <expression>` handler's bound expression (previously shown verbatim, since it shares its line with the handler's own "signature") rather than showing it. Also fixes a latent regex-backend bug this parity work surfaced: a QML `signal foo(...)` declaration has no `{...}` body, but `_def_span_end`'s brace-matcher — lacking a `;` terminator to stop at — walked forward to the next unrelated `{` in the file (e.g. a sibling `MouseArea { ... }` block) and wrongly claimed its lines as part of the signal's span; both call sites (`extract_regex_edges`'s generic loop and `_extract_qml_signal_symbols`, shared by the tree-sitter backend) now give a QML signal a single-line span unconditionally. New `tests/test_read_modes.py` covers both new modes, the new tool's skeleton/full behavior on a large Python file and on the qt_app fixture's C++ header and `.qml` files, ledger savings, default-mode byte-identity, and `_meta` routing. `evals/run_evals.py` precision/recall/tokens are unchanged (skeleton bundle packing's file-selection logic is untouched by this change).
- Add a standard `_meta` envelope to all 8 MCP read/query tools (P1-5): `_meta = {index_age_seconds, indexed_at, fingerprint_fresh, auto_refreshed?, cached?, saved_tokens?}` is now assembled in one shared place (`mcp/tools._build_meta`, called from `_format_payload`) instead of each tool merging staleness status ad hoc. `detailed` responses always carry `_meta`; a `concise` response keeps it only when something is noteworthy — the index is stale, an auto-refresh just ran, a P1-3 cache hit occurred, or `saved_tokens` is positive — and otherwise omits it entirely (0 added tokens), decided by `_record_tool_usage` after the P0-1 ledger computation runs. `saved_tokens` is *never* a second, parallel estimate: it's exactly `baseline_tokens - response_tokens` from the same `_estimate_baseline` call the ledger row records, folded into the already-built `_meta` post hoc so the ledger and the visible number can't drift; negative/zero savings on tiny fixtures are recorded in the ledger as before but never surfaced in `_meta` (that's noise, not a savings claim). Fixes the P1-3 caveat noted in that entry below: the `query_cache` table now stores each tool's bare response DATA only (`CortexStore.get_query_cache`/`set_query_cache`, docstrings updated), never the status/`_meta`-merged result, and `_meta` is rebuilt fresh from a newly computed staleness check after *every* cache lookup — hit or miss — via a new `CortexStore.get_repo_indexed_at` (`repos.updated_at`). A cache hit therefore reports `cached: true` and the CURRENT index age, never a replayed `auto_refreshed` block or an age frozen at write time (new test: `test_cache_hit_shows_current_index_age_not_frozen_write_time_meta` in `tests/test_query_cache.py`, using a mocked clock to prove the age advances across a hit). The separate `bundle.generated_at` timestamp embedded in `cortex_query`'s `detailed` DATA payload is unaffected by this fix and intentionally so — it is retrieval data (when *that* bundle was computed), not response metadata, and `_meta` is additive only; `_meta.indexed_at`/`index_age_seconds` are the fields to trust for "how fresh is the index right now." `_meta`'s `index_age_seconds`/`indexed_at` live only inside the envelope, not duplicated at the top level of `detailed` responses (`_META_ONLY_STATUS_KEYS`); computing them is wrapped in `try/except` at two levels (`_staleness`'s `get_repo_indexed_at` call, `_format_payload`'s `_build_meta` call) so a broken lookup degrades to omitting/zeroing the field, never to breaking the tool response. `_without_meta` strips `_meta` before every token count used for P0-1/baseline purposes (the ledger's `response_tokens`, and `_estimate_baseline`'s detailed-rendering baseline for `cortex_search_symbols`/`cortex_relations`/`cortex_overview`) so the envelope can't inflate the very savings numbers it reports. `evals/run_evals.py` calls `generate_bundle` directly, bypassing `_format_payload` entirely, so eval metrics are unaffected. Updated the P1-3 "byte-identical" cache tests (`tests/test_query_cache.py`) to compare payloads with `mcp_tools._without_meta` (a hit's `_meta.cached` now legitimately differs from a miss's) and updated three P0-1 baseline tests (`tests/test_token_savings.py`) whose expected token counts were computed from a `detailed` rendering that now always carries `_meta`. New `tests/test_meta_envelope.py` covers `_meta` presence in `detailed` mode across all 8 tools, absence in `concise` mode with nothing noteworthy, `saved_tokens` matching the ledger row exactly, and non-fatal degradation on lookup/estimate failures.
- Fix cross-file Qt signal/handler symbol resolution (P0-4): `emit`/`Q_EMIT` and `connect()` in a `.cpp` used to resolve only against symbols in that *same* file, so `emit deviceConnected(42)` produced an `emits` edge to a placeholder `module:deviceConnected` node whenever the signal was declared in the header instead (the normal C++ header/implementation split); QML `onFoo:` handlers (`onClicked:`, `onDeviceConnected:`) never became symbol nodes at all, only a `handles` edge to a placeholder. `structural/regex_backend.py::_extract_qt_cpp_edges`/`_extract_qml_handlers` now tag every `emits`/`connects`/`handles` edge with the referenced name(s) (`signal_name`/`slot_name`/`sender_class`/`receiver_class`/`component_path`), and a new `graph.py::QtSymbolIndex` + `_resolve_qt_edges` pass runs once at the end of `build_file_layer`, after every file in the batch has produced its nodes, to resolve any placeholder against declared signals/slots across the whole batch: first via the emitting file's own `#include`s, then a `sender_class`/`receiver_class`-qualified lookup for `connect()`, then a repo-wide unique-name fallback for `emit`s; a QML handler resolves only when its enclosing instantiated block is a locally known component (`onDeviceConnected` → the `DeviceDelegate` it's declared on), never a same-file guess, so a handler for an external Qt Quick type (e.g. `MouseArea`) correctly stays an unresolved placeholder. Each QML handler is now also its own real symbol node (kind `func`, `qt: handler` metadata), addressable like any other symbol via `cortex_search_symbols`/`cortex_read_symbol`. Also fixes a latent same-file false-positive: `_symbol_ref` previously matched *any* same-named node regardless of Qt tagging, so a `connect()` referencing `&DeviceModel::onDeviceConnected` could silently resolve to an unrelated `DeviceManager::onDeviceConnected` method body living in the same `.cpp` — it now only matches nodes explicitly tagged `qt: signal`/`qt: slot`. Handles the P0-3 incremental-ingest interaction: a signal declared in a header that isn't part of an incremental batch (only the `.cpp` changed) still resolves, via a new `CortexStore.fetch_qt_symbol_index` the incremental path in `ingest.py` consults before its delta delete (and filters to exclude the batch's own stale/deleted paths, so a rewritten or removed file's old declarations can't leak in as a stale resolution target). The regex backend's Qt-specific extraction is reused directly by the tree-sitter backend (as it already was for C++ signals/slots), including a new `_extract_qml_signal_symbols` bridge for QML `signal` declarations, so both backends resolve identically; not independently verified against real `tree_sitter_cpp`/`tree_sitter_language_pack` parses in this change's environment (extras not installed there), but the shared-function design means the tree-sitter path exercises the same Qt logic as the regex path, which is fully tested. No change to existing non-Qt structural extraction or `evals/run_evals.py` precision/recall.
- Add per-kind tokenizer calibration and an optional exact tokenizer extra (P1-4): `count_text_tokens`/`truncate_text_to_budget` (`src/cortex/tokenizer.py`) now take a `kind: "code" | "markdown" | "text"` parameter (default `"text"`) and, on the stdlib heuristic path, scale the existing regex-segment estimate by a per-kind `CALIBRATION` factor instead of treating one segment as one token uniformly. `kind` is threaded through every call site that already has `SourceRecord.kind` available: `bundle.py`'s whole-file and skeleton token counts (`generate_bundle`, `_skeleton_item`) and truncation, `benchmark.py`'s corpus-token sum, and `mcp/tools.py::_call_read_symbol`'s span body (always structural code, so hardcoded to `"code"`). JSON-payload token counts in `mcp/tools.py` (relations/search/overview renderings) and `impact.py`/`references.py` are deliberately left at the `"text"` default since they count formatted JSON, not raw source prose. Added the optional `[tokens]` extra (`tiktoken>=0.7`); when importable, `count_text_tokens` uses a cached `o200k_base` encoder directly for an exact count (mirroring the existing soft `import regex` pattern already in the file) and `kind`/`CALIBRATION` become irrelevant to the result; without it — the default, stdlib-only install — the calibrated heuristic runs, and if `tiktoken` is installed but its encoding data can't be fetched (e.g. restrictive egress), it degrades to the same heuristic rather than raising. `truncate_text_to_budget` was also rewritten from a fixed proportional segment cut to a binary search against `count_text_tokens` itself, so the trimmed output is always bounded by the *same* estimator that computed the budget (tiktoken or heuristic) instead of assuming a 1:1 segment-to-token ratio while slicing. New dev-time-only `evals/calibrate_tokenizer.py` measures `CALIBRATION` by comparing the stdlib heuristic against `tiktoken`'s `o200k_base` encoding over a repo's own sources grouped by kind (not a runtime dependency; degrades with a clear message if `tiktoken` isn't installed). **Caveat**: the `CALIBRATION` constants currently ship as provisional `1.0` (no-op) placeholders, not measured values — the sandbox this change was authored in blocks the network host `tiktoken` fetches its encoding data from, and a hash-verified third-party mirror of that same file was correctly declined as a source of truth for production constants by this change's security review. The full mechanism (kind threading, tiktoken integration, calibration script, binary-search truncation) is implemented and tested; re-run `pip install tiktoken && python3 evals/calibrate_tokenizer.py` in an environment with normal internet access and paste the printed factors into `CALIBRATION` to complete the calibration. `evals/run_evals.py` precision/recall are unchanged from pre-P1-4 (1.0 factors keep the heuristic path behaviorally identical); one per-task token count moved by <1% from the truncation rewrite alone, with no effect on any precision/recall/file-selection metric.
- Add a read-through result cache for `cortex_query`/`cortex_impact`/`cortex_overview` keyed on repo fingerprint (P1-3): a new `query_cache` table (`CortexStore.get_query_cache`/`set_query_cache`/`prune_query_cache`) stores each tool's full response under `cache_key = sha256(fingerprint + tool + canonical_json(args))` (`mcp/tools._cache_key`), computed from the fingerprint `_ensure_fresh` settles on *after* any auto-refresh so a stale-then-refreshed call never caches under an already-wrong fingerprint. A repeated identical call is a single SQLite `SELECT` and skips PageRank/packing (or report generation, or impact ranking) entirely; any file change invalidates automatically, since it changes the fingerprint the key is built from. Cache reads/writes are wrapped in `try/except` so a broken cache degrades to recompute, never to a broken response; error responses are never cached. `CORTEX_QUERY_CACHE=0` disables both reads and writes. `cortex gc` now also prunes each repo's cache (rows older than 30 days, then oldest-first beyond 200 rows/repo) unconditionally, alongside its existing `--prune`-gated orphaned-dir cleanup. Cache hits still record to the P0-1 token-savings ledger (via the existing `call_tool` → `_record_tool_usage` hook, unchanged) since the token savings are real to the caller whether or not Cortex recomputed. MCP response shape is unchanged; the `_meta.cached` flag is deferred to P1-5. Caveat: `cortex_query`'s `response_format: "detailed"` payload embeds `generated_at` (a build-time timestamp) from the underlying bundle, so a cache hit echoes the timestamp from when the entry was first written rather than the current time — this is inherent to caching a timestamped field and does not affect the default `concise` format, which excludes it.
- Add a SQLite FTS5 full-text layer with rank fusion (P0-2): a new `source_fts` virtual table (`tokenize='unicode61'`, plus an auxiliary `identifiers` column reusing the existing camelCase/snake_case splitter so both `deviceConnected` and `device connected` match) is kept in sync inside `save_sources`/`delete_sources`/`reset_repo`, guarded by a runtime FTS5-availability check (`CortexStore._init_fts5`/`fts_enabled`) that falls back cleanly to the existing LIKE-based search path when FTS5 isn't compiled into a given Python's sqlite3 build, including a one-time backfill for databases created before this change. `CortexStore.search_fulltext` returns `(path, bm25, line-anchored snippet)` results. New MCP tool `cortex_search_text(query, limit, budget)` is a grep replacement over indexed file bodies — string literals, error messages, comments, Markdown prose — that `cortex_search_symbols` (name/signature/path only) can't reach. `generate_bundle`'s ranking now fuses its existing name/keyword score with the new FTS list (plus a definition-boost list) via a new generic `src/cortex/fusion.py::rrf_fuse` helper (reciprocal rank fusion over N ranked lists, so an optional future embedding retriever can be added as one more list without another ranking rewrite); an identifier-shaped query (camelCase/snake_case/`::`-qualified) adaptively double-weights the lexical/name list over FTS body hits, and the fusion contribution is scaled well below `NAME_MATCH_BONUS`/`PATH_MATCH_BONUS` so an exact stem/symbol hit still wins. Docs (`README.md`, `skills/cortex/SKILL.md`, `hooks/session-start.py`) updated for the new tool.
- Add a token-savings ledger (P0-1): every successful MCP read-tool call (`cortex_query`, `cortex_overview`, `cortex_impact`, `cortex_search_symbols`, `cortex_read_symbol`, `cortex_relations`, `cortex_references`) now records its actual response tokens and a deterministic baseline estimate to a new `tool_usage` table (`CortexStore.record_tool_usage`/`fetch_tool_usage`), via one auditable `_estimate_baseline` helper in `mcp/tools.py`. File-returning tools baseline against the raw content of every distinct referenced file; `cortex_search_symbols`/`cortex_relations`/`cortex_overview` baseline against their own `detailed`-format rendering (plus referenced files for `cortex_relations`). Ledger writes are wrapped so a failure never breaks or alters the underlying tool response. Add `cortex saved [repo] [--daily] [--format text|json] [--price-per-mtok in,out]` (`src/cortex/savings.py`) to report totals, per-tool and daily rollups, and optional dollar figures at a user-supplied rate. MCP response shapes are unchanged; the `_meta.saved_tokens` envelope is deferred to P1-5.

## 0.7.3 — 2026-07-08

- Boost whole-identifier matches in `cortex_search_symbols` over sub-token floods. A camelCase/snake query splits into tokens matched independently, so a query containing one common token ("flow") could flood the results — and, via the candidate `LIMIT`, drop the real target entirely — with that token's matches. Candidate fetch now orders by a whole-identifier `LIKE` pattern (tokens in sequence, so `DeviceListModel` / `device_list_model` / `device list model` all match either spelling) and then by how many query tokens hit the label; ranking adds a whole-query-embedded tier above scattered-token matches and grades partial name matches by token coverage.
- Weight directory-path tokens in `cortex_query` ranking (`PATH_MATCH_BONUS = 40`): a task term matching a path segment ("ui", "backend", "mcp") now breaks ties between same-named files, weaker than the existing stem/symbol `NAME_MATCH_BONUS` but stronger than body keyword density.
- Return multiple ranked snippets from `cortex_query` instead of one budget-filling file dump: when more than one candidate matches, each item is capped at `ITEM_BUDGET_SHARE = 0.4` of the budget (oversized items degrade to skeleton/truncated form via the existing packing fallbacks), so the top file can no longer crowd out every other match.

## 0.7.2 — 2026-07-08

- Fix `cortex_search_symbols` (P0) burying the queried symbol under alphabetical, unrelated results. `CortexStore.search_nodes` matched query tokens against each node's `source_ref` (file path) with the same weight as the symbol name, so a query like `CortexStore` matched the *path* of every symbol under `src/cortex/` and ranked `__init__`, `_bfs_proximity`, … equal to the real class (then sorted alphabetically). With no `ORDER BY`, the genuine node could also be dropped by the candidate `LIMIT` before ranking ran. Now the SQL fetches label matches first (`ORDER BY` a label/signature/path priority) so a name match is never truncated away, and the Python ranker buckets strictly name → signature → path, so path-only hits always rank last. (Not related to `enrichment_enabled`, which does not gate symbol matching.)
- Fix `cortex_read_symbol` (P1) returning a single declaration line for C/C++/QML/JS/… symbols. The regex structural backend hard-coded `span_end = span_start`; it now brace-matches the body (skipping strings and comments) to the closing brace, and anchors the line number and signature on the symbol *name* so leading `^\s*` / greedy return-type matches no longer skew the span onto a blank or preceding line. Also fixes `_signature` overshooting toward EOF for a symbol on the final line with no trailing newline.
- Fix `cortex_impact` (P2) listing commit SHAs as impacted paths ranked `1.0`. Commit nodes default to `granularity="file"` and link to files via COCHANGE `touches` edges, so `rank_file_impact` counted them as file neighbors; it now excludes commit nodes with the same guard the community/rank layers use.
- Improve `cortex_query` relevance (P2) when the task names a language or extension: same-language files are boosted and other code languages demoted (multiplicatively, so unrelated files are never seeded by language alone), so e.g. a QML task no longer resolves to the same-named C++ file.

## 0.5.1 — 2026-07-07

- Slim `cortex_relations` and `cortex_impact` output: edge endpoints collapse to a single `"label @ path:line"` string instead of a 3-field object; internal-only `edge_id`/`layer` fields dropped from both tools' responses (`cortex_relations` also drops `weight`/`confidence`, kept in `cortex_impact`'s `why` since it drives ranking there). Both tools now honor a token `budget` param (default 2000, same mechanism as `cortex_query`'s bundle), returning `truncated`/`returned_count` so a broad query can't dump hundreds of edges into a caller's context every turn.

## 0.5.0 — 2026-07-07

- Add `cortex_relations` MCP tool: query graph edges filtered by relation type (`contains`, `imports`, `inherits`, `emits`, `connects`, `handles`) at symbol granularity, e.g. "who inherits class X", "who emits signal Y". Previously the C++/QML/Qt structural edges added in 0.3.0–0.4.1 had no query path. Backed by new SQL-filtered `CortexStore.query_edges()`/`get_nodes()` (no full-graph load); output resolves edge endpoints to labels/paths without dumping raw edge metadata.

## 0.4.1 — 2026-07-07

- Fix: C++/QML structural extraction never produced `inherits` edges. Regex backend's base-clause capture was discarded; tree-sitter backend had no inheritance extraction at all. Both backends now emit `inherits` edges for `class Foo : public Bar`-style declarations.
- Fix: Qt signal/slot/emit/connect/Q_OBJECT detection previously existed only in the regex fallback, so it silently vanished for any file the tree-sitter backend successfully parsed. Tree-sitter path now runs the same Qt/QML detection regardless of which backend handles the file.

## 0.4.0 — 2026-07-07

- Per-repo data (`cortex.db`, `cortex_report.md`) now lives in a central store at `~/.cortex/data/<sha256-prefix-of-repo-path>/` instead of a `.cortex/` directory inside the target repo. Indexing no longer touches the target repo.
- Existing in-repo `.cortex/` directories keep working (legacy fallback); delete one to migrate that repo to the central store on next refresh.
- New: `CORTEX_DATA_DIR` env var overrides the central store location.
- New: `cortex gc [--prune]` lists or deletes central data dirs whose source repo is gone.
- Each central data dir carries a `meta.json` recording the source repo path.

## 0.2.3 - 2026-07-07

Auto-refresh: the index now keeps itself current — no manual `cortex_refresh` discipline needed.

- MCP read tools (`cortex_query`, `cortex_overview`, `cortex_impact`, `cortex_search_symbols`) detect a stale fingerprint and run an incremental ingest before answering, so results always reflect the working tree. Responses report what changed under `auto_refreshed`. Disable with `CORTEX_AUTO_REFRESH=0`.
- Missing-database behavior is unchanged: read tools still error with a `cortex_refresh` hint rather than building an index implicitly.
- Incremental ingest now removes deleted files from the index (`deleted_files` in the summary) instead of leaving ghost sources behind.
- Fixed stale graph rows surviving incremental ingest: graph saves during incremental runs now replace the repo graph wholesale instead of upserting, so symbols removed from a file disappear from search results.

## 0.2.2 - 2026-07-06

Retrieval-quality fixes from a live efficacy eval on the Cortex repo itself, where a bundle for an implementation question returned docs plus a stale `build/` duplicate and missed the real source file.

- Ingest now honors `.gitignore` (via `git ls-files --cached --others --exclude-standard`) and skips common artifact dirs (`build`, `dist`, `dist-check`, `.venv`, `venv`, `.tox`, `.eggs`, `.ruff_cache`) so stale copies never enter the graph. Repo fingerprints use the same file listing.
- Bundle scoring adds a name-match bonus: a task term that exactly matches a file stem or symbol name now outranks keyword-dense docs.
- Markdown items are capped at 40% of the bundle budget whenever code candidates also match, so docs can't crowd out implementation files. Docs-only repos are unaffected.
- SessionStart hook stays silent outside git repositories instead of nudging about a missing index.
- New `noisy_lib` eval fixture (fat README, doc plans, gitignored `build/` duplicate) with implementation-detail gold tasks to catch this failure class in CI.
- All manifest versions aligned at 0.2.2; manifest test now asserts `hooks/hooks.json` is the single source of hook wiring.
